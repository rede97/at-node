# CH582 低功耗设计指南

> 基于 AT-Node 项目实践，总结 CH582 系列 MCU 的低功耗架构原理与工程策略。

## 1. 核心原理：时钟域分离

CH582 低功耗的基础是在物理上把时钟分成两个独立的域：

| 时钟域 | 频率 | 典型功耗 | 睡眠时 |
|--------|------|----------|--------|
| 高速域（HSE + PLL + CPU + Flash） | 32 MHz → 60 MHz | 2–5 mA | **全部关断** |
| 低速域（LSI/LSE + RTC） | 32 kHz | 1–2 µA | **持续运行** |

高速域处理 BLE 收发、AT 命令、按键扫描。低速域只做一件事：在正确的时间唤醒高速域。两个域可以在物理上独立供电控制——电源隔离是 CH582 睡眠逻辑的硬件基础。

如果没有低速域，必须留一颗定时器在高速域上保持计时。但这颗定时器依赖高速时钟，高速域关断时它也停了——要么不深度睡眠（mA 级持续耗电），要么睡了永远醒不来。

## 2. TMOS：统一睡眠决策

TMOS 本质上是一个**事件驱动的定时器汇总器**——所有软件任务把自己的"下次到期时间"注册到 TMOS，TMOS 遍历队列取最近那个时刻，写进 RTC 比较寄存器，然后关断一切进入睡眠。

```
hws_key_poll      → 下次 100ms 后
at_parser_poll    → 下次 10ms 后
ble_calibration   → 下次 120s 后
ble_connection    → 下次 10ms 后   （BLE 栈内部）

TMOS 取 min = 10ms → RTC 触发设在 10ms → 睡眠
```

### 2.1 为什么是协作式而非抢占式

抢占式 RTOS 需要 Tick 中断来驱动上下文切换。即使没有任务就绪，Tick 也要定期触发 CPU 醒来检查状态。每次醒来意味着高速域重新上电、晶振起振、PLL 锁定——这个开销在 CH582 上约 1.4 ms。

在 30 ms 连接间隔下，1 kHz Tick 会让 CPU 每秒白醒 1000 次，其中 ~967 次是空的。而 RTC 统一调度一次只醒一次。

差别不在醒着的 1 ms 干了什么，在于那 29 ms 里**高速域是断电还是待命**——2 µA vs 3 mA，差 1500 倍。

### 2.2 TMOS vs FreeRTOS 的低功耗差异

| | TMOS | FreeRTOS |
|---|---|---|
| 调度触发 | RTC 比较器（只在有事时醒） | SysTick 中断（固定频率） |
| Tickless idle | 天然（定时器队列即睡眠决策） | 需要额外实现 `vPortSuppressTicksAndSleep()` |
| BLE 集成 | 同一定时器队列（BLE 连接事件就是 TMOS 事件） | BLE 事件在 RTOS 外部，RTOS 不知道什么时候需要醒 |
| 实现复杂度 | 状态机扫描器 + 定时器链表 | 完整 RTOS 内核 |

## 3. 睡眠模式分级

### 3.1 硬件睡眠模式

| 模式 | 高速域 | CPU | SRAM | RTC | 唤醒后恢复时间 | 典型电流 |
|------|--------|-----|------|-----|---------------|----------|
| **Idle** | 运行 | 暂停（WFI） | 保持 | 运行 | <1 µs | ~100 µA |
| **Sleep** | 关断 | 关断 | 部分保持 | 运行 | ~1.4 ms | ~2 µA |
| **Shutdown** | 关断 | 关断 | 可配置 | 可选 | ~2 ms | ~0.6 µA |

CH582F 的 `CH58X_LowPower()` 实现的是 Sleep 模式——RTC 触发唤醒，高速域重新上电。
进入 Shutdown 需要额外配置，目前 AT-Node 未启用。

### 3.2 应用层功耗分级（以 BLE 输入设备为例）

借鉴成熟 BLE 鼠标的策略，将功耗按用户活跃度分为多级：

```
Level 0  — 活跃操作          连接间隔 7.5–10 ms   Latency 0      传感器全速    5–10 mA
Level 1  — 短暂静止 (<1s)    连接间隔 10 ms        Latency 0      传感器保持    3–5 mA
Level 2  — 静止 (<10s)       连接间隔扩至 50 ms    Latency 1–3    传感器降速    ~500 µA
Level 3  — 长时间静止 (<5min) 连接间隔 100 ms       Latency 9      传感器关闭    ~200 µA
Level 4  — 深度睡眠 (>5min)   断开 BLE 连接         RTC 触发唤醒    运动传感器唤醒  ~5 µA
```

每一级的升级由**无活动计时器**触发，降级由**输入事件**（按键、传感器）触发。

Level 2 和 Level 3 是通过 GAP Connection Parameter Update Request 实现的——Peripheral 向 Central 请求改变连接间隔和 Slave Latency。这是 BLE 标准机制，不需要重新配对。

### 3.3 当前 AT-Node 的实现状态

| 特性 | 状态 | 位置 |
|------|------|------|
| HWS_SLEEP 编译开关 | ✅ | `config.h` |
| RTC 唤醒睡眠 | ✅ | `hws_sleep.c → hws_sleep_enter()` |
| 睡眠 GPIO 漏电防护 | ✅ | `hws_platform_init()` — GPIO 全设输入上拉 |
| BLE 温度校准（防睡眠漂移） | ✅ | `hws_core.c → HWS_CALIB_EVENT`（每 120s） |
| 固定连接参数 | ✅ | `hidkbd_ble.c` — 10ms / Latency 0 |
| **动态连接参数调整** | ❌ | 未实现 — 连接后始终保持最快参数 |
| **分级睡眠状态机** | ❌ | 未实现 — 只有全速/睡眠两个状态 |
| **运动/按键中断唤醒** | ❌ | 未实现 — 只有 RTC 定时唤醒 |

### 3.4 后续优化方向

**短期**（最小改动）：
- 连接参数更新分两级：活跃时 10ms/Latency 0，静止 2s 后扩至 50ms/Latency 3
- 利用已有的 `START_PARAM_UPDATE_EVT` 事件机制

**中期**（增加睡眠分级）：
- 无活动 10s → Level 3（关闭传感器/LED，连接间隔进一步拉大）
- `hws_led_update()` 增加约束：Level ≥ 3 时禁用 blink

**长期**（深度睡眠）：
- 无活动 5min → Level 4（通过 AT 命令或长按按键退出）
- 需实现 GPIO 边沿唤醒（目前只支持 RTC 唤醒）
- 唤醒后需重新执行 `hws_platform_init()` + `hws_init()` 恢复硬件状态

## 4. BLE 连接参数与功耗的定量关系

### 4.1 连接间隔

| 连接间隔 | 每秒连接事件 | 活跃时间占比 | 平均电流 |
|----------|-------------|-------------|---------|
| 7.5 ms | 133 | 13.3% | ~800 µA |
| 10 ms | 100 | 10% | ~600 µA |
| 30 ms | 33 | 3.3% | ~200 µA |
| 100 ms | 10 | 1% | ~60 µA |
| 1000 ms | 1 | 0.1% | ~10 µA |

活跃时间按每次连接事件 CPU 工作 1 ms 估算。实际值依赖 CPU 处理的数据量。

### 4.2 Slave Latency

Latency = N 表示每 (N+1) 个连接事件只响应 1 次：

```
10 ms 间隔 + Latency 0 → 每秒 100 次响应
10 ms 间隔 + Latency 9 → 每秒 10 次响应（跳过 90% 的连接事件）
```

Central 端仍以为连接活着（空包自动应答），Peripheral 端跳过大部分事件不干活。

### 4.3 响应延迟 vs 功耗

| 配置 | 最坏报告延迟 | 平均电流 |
|------|------------|---------|
| 7.5ms / Lat 0 | 7.5 ms | ~800 µA |
| 10ms / Lat 0 | 10 ms | ~600 µA |
| 10ms / Lat 4 | 50 ms | ~200 µA |
| 30ms / Lat 0 | 30 ms | ~200 µA |

**关键认知**：延迟增加 5–6 倍，功耗降低 3–4 倍。输入设备的功耗优化本质上是在"用户感知度"和"电池寿命"之间换空间。键盘按键可以被 50 ms 延迟感知到，鼠标移动不行——所以鼠标在活跃时必须保持 ≤10 ms，但静止后可以立刻降速。

## 5. 硬件平台对比

### 5.1 Nordic nRF52 的做法

nRF52 的 BLE Radio 有**硬件状态机**——可以在 CPU 完全睡眠的情况下自动处理连接事件的收发、应答、跳频。只有当应用层有数据需要处理时才中断唤醒 CPU。

```
nRF52:  Radio HW → 自主处理 BLE → 有数据才叫 CPU
CH582:  RISC-V CPU → 跑 BLE 软件栈 → 每个连接事件都醒 CPU
```

Nordic 的 PPI（Programmable Peripheral Interconnect）系统允许外设之间直接通信不经过 CPU。这进一步减少了 CPU 唤醒频率。

CH582 没有硬件 BLE 状态机和 PPI，所以 CPU 必须在每个连接事件中醒过来处理 BLE 软件栈。TMOS 的价值在于：尽管必须醒，但**只需醒一次，不会白醒**。

### 5.2 成本权衡

| | CH582 方案 | Nordic 方案 |
|---|---|---|
| 芯片成本 | $0.50 | $2.50+ |
| BLE 处理 | CPU 软件 | Radio 硬件 |
| 低功耗机制 | TMOS（软件汇总） | PPI（硬件路由） |
| 开发复杂度 | 理解 TMOS 状态机 | 理解 nRF Connect SDK + Zephyr |
| 极限功耗 | ~2 µA（Sleep） | ~0.3 µA（System OFF） |
| 典型应用场景 | 低功耗外设，成本敏感 | 超低功耗，功能复杂 |

两种设计到达同样的终点——"汇总所有未来事件，睡到最近的那个"——但实现路径不同。CH582 用软件弥补硬件缺位，Nordic 用硬件降低软件复杂度。

## 6. 设计原则（供添加新功能时参考）

1. **功耗是设计约束，不是事后优化。** 任何新加的定时器都要回答：周期多少、能不能和现有周期对齐、能不能在静止时拉长。

2. **TMOS 定时器对齐。** 新增周期性任务的周期应取现有周期的约数/倍数（10 ms / 100 ms / 120 s），避免制造无法对齐的额外唤醒。

3. **添加外设前先加 `#ifdef HWS_*` 开关。** 未使用的模块在编译期移除，节省 Flash 和 TMOS 定时器槽位。

4. **运动/按键中断唤醒 > RTC 周期唤醒。** 外部中断允许 CPU 在完全断电的状态下被唤醒（不需要定时器），是最省电的唤醒方式。

5. **BLE 连接参数应与用户活跃度联动。** 不动的设备不需要 10 ms 连接间隔。分级策略参考 §3.2。

6. **每次测量，不要猜测。** CH582 实际功耗受 DC-DC 效率、GPIO 漏电、PCB 布局、外部电路（LDO/上拉/LED）的共同影响。用万用表在电池回路中实测，不要相信估算。
